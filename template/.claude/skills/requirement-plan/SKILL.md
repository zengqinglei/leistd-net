# 需求分析 (requirement-plan)

> 解析自然语言需求 → 生成五段闭环 Plan.md 草案

---

## 执行前必读

- 先读项目规范，确定文档结构和命名规范。
- 使用五段闭环结构生成 Plan.md。
- 必须包含验收闭环，不能只写验收标准。
- 每个 Must have 必须映射到实施步骤和验证方式。
- 不贴大段代码，只放关键结构、接口、目录树、架构图。

---

## 工作流程

```
1. 读取项目规范（如有）
   → 优先读取 `docs/standards/agent-workflow.md` 作为项目级索引
   → 按索引读取必要的命名、需求、技术栈和模块文档
   → 索引不存在时再扫描 `docs/standards/` 常见路径

2. 无项目规范时
   → 使用本 Skill 默认五段闭环模板
   → 提示用户建议创建 `docs/standards/agent-workflow.md`

3. 解析用户需求
   → 背景：为什么做
   → 需求：做什么、谁使用、核心场景
   → 目标：交付结果
   → 范围：本次包含 / 不包含
   → 策略：总体策略、目录树、架构图、注意事项
   → 实施：步骤清单、文件变更清单、执行顺序
   → 风险：风险、概率、影响、应对策略、触发后的处理
   → 验收：Must have / Nice to have、验收映射、完成定义

4. 生成 Plan.md
   → 保存到规范指定路径
   → 默认路径：docs/requirements/req-yyyymmdd-nnn-plan.md
   → 文件名使用 lowercase kebab-case
   → 标题中的需求 ID 保持 REQ-YYYYMMDD-NNN 大写

5. 质量检查
   → 读取生成的文件
   → 对照下方质量检查清单逐条检查
   → 输出检查报告
```

## 五段闭环结构

需求 / Plan 文档必须使用五段闭环结构：

### `## 1. 需求&背景&目标`

- 背景：为什么做
- 需求：做什么、谁使用、核心场景是什么
- 目标：交付结果
- 范围边界：本次包含 / 本次不包含

### `## 2. 核心策略`

- 总体策略
- 调整内容树形目录结构
- 核心架构图
- 重点注意事项及策略

### `## 3. 实施步骤`

- 步骤清单（含验证方式）
- 文件变更清单
- 执行顺序

### `## 4. 风险及应对策略`

- 风险 / 概率 / 影响 / 应对策略 / 触发后的处理

### `## 5. 验收闭环`

- 验收标准：Must have / Nice to have
- 验收映射：验收项 / 对应实施步骤 / 验证方式 / 证据
- 完成定义：代码、审查、测试、构建、部署、用户验收

---

## 生成规则

### 范围定义

**必须识别**：
- 背景：当前问题、业务背景、为什么现在做。
- 需求：功能描述、用户角色、使用场景。
- 目标：交付结果、衡量标准。
- 范围：本次包含、本次不包含。

**格式**：
```markdown
### 1.1 背景
说明当前问题和业务背景。

### 1.2 需求
说明要做什么、谁使用、核心场景。

### 1.3 目标
- 目标 1
- 目标 2

### 1.4 范围边界

#### 本次包含
- 范围 1

#### 本次不包含
- 范围 1
```

### 核心策略

**必须包含**：
- 总体策略：怎么做、为什么这样做。
- 目录树：涉及文件和目录的树形结构。
- 架构图：核心数据流或组件关系。
- 注意事项：风险点、兼容性、技术约束。

### 实施步骤

**每个步骤必须包含**：
- 步骤编号
- 实施内容
- 产出物
- 验证方式

**必须包含文件变更清单**：

```markdown
| 文件 | 操作 | 说明 |
|------|------|------|
| path/to/file | 新增/修改/删除 | 说明 |
```

### 验收标准

**Must have**（必须满足）：
- 每个 Must have 对应一个可验证的功能点。
- 每个 Must have 必须映射到至少一个实施步骤。
- 每个 Must have 必须有验证方式。
- 至少 3 个，不超过 7 个。

**Nice to have**（可选增强）：
- 提升体验但不是必需的。
- 可以延后实现。
- 0-3 个。

### 验收映射

```markdown
| 验收项 | 对应实施步骤 | 验证方式 | 证据 |
|------|------------|---------|------|
| 验收项 1 | 步骤 1 | 测试 / 构建 / 页面验证 / API 验证 | 测试报告 / 截图 / 日志 / URL |
```

### 完成定义

```markdown
- [ ] 代码实现完成
- [ ] code-review 无 P0 问题
- [ ] 测试通过
- [ ] 构建通过
- [ ] 部署成功
- [ ] 验收标准全部对照完成
```

### 风险评估

**必须识别**：
- 技术风险（API / 框架 / 性能）
- 资源风险（时间 / 人力 / 预算）
- 外部依赖风险（第三方服务）
- 数据风险（迁移 / 兼容性）

**格式**：
```markdown
| 风险 | 概率 | 影响 | 应对策略 | 触发后的处理 |
|------|------|------|---------|------------|
| 风险 1 | 高/中/低 | 高/中/低 | 应对策略 | 回退 / 降级 / 暂停 |
```

## 质量检查

生成 Plan.md 后，自动检查：

### 必须项
- ✅ 五段结构完整（1-5 各段齐全）
- ✅ 背景、需求、目标清楚
- ✅ 范围边界明确（本次包含 / 不包含）
- ✅ 核心策略包含目录树和架构图
- ✅ 实施步骤可执行（有步骤清单、文件变更清单、执行顺序）
- ✅ 风险有应对策略和触发后处理
- ✅ 验收标准包含 Must have（≥3 个）
- ✅ 每个 Must have 已映射到实施步骤和验证方式
- ✅ 文档无明显重复冗余

### 建议项
- ✅ 有 Nice to have 验收标准
- ✅ 完成定义完整
- ✅ 文档风格符合项目规范

## 后续上下文

- 任务管理上下文：从 `## 3. 实施步骤` 拆任务，从 `## 5. 验收闭环` 提取验收项。
- 开发上下文：对照 `## 2. 核心策略` 和 `## 3. 实施步骤` 实施。
- 审查上下文：检查 `## 5. 验收闭环` 的 Must have 是否漏实现。
- 测试上下文：按 `## 5. 验收闭环` 输出测试覆盖表。
- 部署上下文：部署报告必须输出验收闭环对照。
