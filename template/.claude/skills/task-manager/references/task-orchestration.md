# 任务编排参考

> 仅在需求包含多个子任务、存在明确依赖或用户要求编排时读取。

## 触发条件

- 需求包含 2 个以上子任务。
- 子任务之间存在顺序、依赖或验收映射。
- 用户已确认任务清单或要求继续自动编排。

## 编排流程

1. 从 Plan.md `## 3. 实施步骤` 提取子任务。
2. 结合 `## 2. 核心策略` 识别模块边界。
3. 从 `## 5. 验收闭环` 提取 Must have 并映射到子任务。
4. 标注依赖、执行顺序和每个子任务的验证方式。
5. 向用户展示清单，等待确认。
6. 每次只执行一个子任务，完成后更新上下文并继续下一个。
7. 任一子任务失败时暂停编排，报告原因和建议。
8. 全部完成后输出验收对照表并更新 registry。

## 标准流水线

| 阶段 | 子任务 | Skill | 完成后衔接 |
|------|--------|-------|------------|
| 1. 开发 | 代码实现 + 规范审查 + 构建验证 | coding + code-review | 测试 |
| 2. 测试 | 运行测试套件 + 覆盖率报告 | test-runner | 部署或修复 |
| 3. 部署 | 构建镜像 + 部署 + 健康检查 | deploy | 验收或回滚 |
| 4. 验收 | 汇报用户 + 等待验收 | task-manager | 流程结束 |

## 失败处理

- 先分析根因，不要默认回退到阶段 1。
- 若是当前阶段配置/环境问题，优先修复当前阶段。
- 若需求不明确、方案变化或高风险操作需要确认，等待用户决策。
- 报告格式：失败原因、影响范围、建议动作、是否可重试。
