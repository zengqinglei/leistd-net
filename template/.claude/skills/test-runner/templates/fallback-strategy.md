# 降级策略 (Fallback Strategy)

> 当项目缺少测试配置时的降级行为

---

## 🎯 核心原则

### 1. 优雅降级

- ✅ **有配置**：按项目配置运行测试
- ✅ **无配置**：使用通用测试框架和默认配置
- ❌ **不报错停止**：继续执行，但告知用户

### 2. 提示而非阻塞

- ✅ **提示一次**：首次检测到缺失时提示
- ✅ **继续执行**：使用默认配置完成测试
- ✅ **提供指引**：告诉用户如何创建测试配置

---

## 📋 降级场景

### 场景 1：无测试配置文件（.NET）

**检测**：
```markdown
尝试读取：
1. .runsettings
2. xunit.runner.json
3. testsettings

结果：都不存在
```

**降级行为**：
```markdown
1. 使用 dotnet test 默认配置
2. 提示用户：
   "⚠️ 检测到项目未配置测试运行设置，已使用默认配置运行测试。
   建议创建 .runsettings 以声明项目特定的测试配置。
   参考模板：templates/runsettings-template（本 Skill 提供）"
3. 继续执行，使用默认配置运行测试
```

---

### 场景 2：无测试配置文件（Node.js）

**检测**：
```markdown
尝试读取：
1. vitest.config.ts
2. vitest.config.js
3. jest.config.js
4. package.json 中的 test 配置

结果：都不存在
```

**降级行为**：
```markdown
1. 使用 Vitest/Jest 默认配置
2. 提示用户：
   "⚠️ 检测到项目未配置测试运行器，已使用 Vitest 默认配置。
   建议创建 vitest.config.ts 以声明项目特定的测试配置。
   参考模板：templates/vitest-config-template.ts（本 Skill 提供）"
```

---

### 场景 3：无覆盖率门槛配置

**降级行为**：
```markdown
1. 使用默认门槛：
   - 行覆盖率：≥80%
   - 分支覆盖率：≥70%
2. 提示用户：
   "⚠️ 检测到项目未配置覆盖率门槛，已使用通用标准（行覆盖率≥80%）。
   建议创建 .runsettings 或 vitest.config.ts 声明项目特定的覆盖率要求。
   参考模板：templates/coverage-thresholds-template.md（本 Skill 提供）"
```

## ✅ 最佳实践

### 1. 首次提示时机

- ✅ **首次检测到缺失时**：立即提示
- ✅ **只提示一次**：避免重复打扰
- ✅ **提供解决方案**：告诉用户如何创建

### 2. 提示方式

- ✅ **友好语气**：建议而非命令
- ✅ **明确路径**：告诉用户具体文件路径
- ✅ **提供模板**：给出参考模板路径

### 3. 降级执行

- ✅ **使用默认配置**：确保测试能继续
- ✅ **记录日志**：在测试报告中记录降级行为
- ✅ **可追溯**：用户后续可以补充配置文件
