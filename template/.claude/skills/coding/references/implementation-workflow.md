# 实施工作流参考

> 仅在复杂功能、跨模块变更或需要输出实施方案时读取。

## 实施方案模板

```markdown
## 需求理解
## 接口契约
- URL / Request / Response / Error
## 服务端调整方案
- 模型 / 服务 / 接口 / 数据迁移
## 客户端调整方案
- 路由 / 页面 / 组件 / 状态 / API
## Mock 或测试数据调整
## 文件变更清单
## 实施顺序
## 风险点与验证方式
## 待确认问题
```

## 通用工程原则

- 保持与现有目录、命名、错误处理、日志和测试风格一致。
- 没有项目规范时，不假定语言、框架、组件库或版本。
- 输入校验、权限、安全边界、事务和幂等性要明确。
- 删除/迁移代码必须有依据；影响外部接口或数据时先确认。

## 内置规范审查

1. 读取项目规范附录的检查清单。
2. 对照本次变更逐项标记通过/违规。
3. P0 必须修复，P1 尽量修复，P2 可记录。
4. 用户要求“代码审查”或 PR 前审查时，由后续上下文触发对应代码审查能力。

## 最小验证

- 使用项目声明的构建、类型检查或最小相关测试命令。
- 前端可在必要时启动项目验证主要路径。
- 完整测试矩阵由后续上下文触发对应测试能力。
