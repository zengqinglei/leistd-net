# 代码开发 (coding)

> 按 Plan.md 实施代码开发

## 🚨 执行前必读

- ✅ **直接操作文件**：使用 `read/write/edit` 工具，不调用脚本
- ✅ **读取项目规范**：优先读取项目内的规范文档
- ⚠️ **无规范用默认**：无规范时使用本 Skill 提供的技术栈模板
- ⚠️ **降级提示**：当项目无规范文档时，**必须提示用户**建议创建规范文档
- ✅ **方案先行**：复杂功能先输出方案，用户确认后再实施
- ✅ **官方文档**：使用第三方组件前查阅官方文档

---

## 安全边界

- ✅ **不生成硬编码密码、Token、API Key** — 使用环境变量或配置
- ✅ **不删除现有代码** — 只添加或重构，除非用户明确要求删除
- ✅ **不修改 .gitignore / CI 配置** — 除非用户明确要求
- ✅ **不生成测试数据到生产库** — 测试数据放在 `tests/` 或 `mocks/`

---

## 📋 工作流程

```
1. 读取 Plan.md（五段闭环结构）
   → `## 1. 需求&背景&目标`：确认范围和目标
   → `## 2. 核心策略`：确认架构、目录、注意事项
   → `## 3. 实施步骤`：确认执行顺序和文件清单
   → `## 4. 风险及应对策略`：识别开发中需注意的风险点
   → `## 5. 验收闭环`：确认必须满足的验收项

2. 读取项目规范
   → 扫描 `docs/standards/code-standard/` 目录，读取所有规范文档
   → 如无规范，使用本 Skill 模板

3. 输出实施方案
   → 接口契约
   → 后端调整方案
   → 前端调整方案
   → 目录结构
   → 用户确认

4. 实施代码
   → 后端代码
   → 前端代码
   → Mock 数据

5. 规范审查（强制，见下方详细说明）
   → 读取项目规范附录检查清单
   → 逐条对照本次代码变更
   → 有 ❌ 先修复再进入验证

6. 验证（构建检查）
   → 构建检查（dotnet build / ng build）
   → 类型检查（编译通过）
   → 浏览器验证（前端：启动项目验证主要路径）
   
**注意**：测试运行（dotnet test / ng test）由 Phase 4 的 `test-runner` Skill 统一执行，本 Skill 不运行测试。
```

## 🎯 核心能力

### 1. 读取项目规范

**操作**：
```markdown
使用 `read` 工具扫描 `docs/standards/code-standard/` 目录，读取所有规范文档。

如无规范文档：
- 使用本 Skill 模板：`templates/techstack-dotnet-angular.md`
- 提示用户创建项目规范文档
```

**提取信息**：
```markdown
从规范文档中提取：
- 分层架构规则
- 编码规范（record/主构造函数/namespace 等）
- 命名规范（DTO/Service/Component 等）
- 检查清单（逐条可勾选的规范项）
```

---

### 2. 输出实施方案

**时机**：复杂功能、跨模块调整、用户要求先审查

**输出格式**：
```markdown
## 需求理解
## 接口契约
- URL: GET/POST ...
- Request/Response
## 后端调整方案
- Entity
- DomainService
- AppService
- Controller
- DTO
## 前端调整方案
- Route
- Page
- Widgets
- Service
- DTO
## Mock 调整方案
## 调整内容树形目录结构
## 核心代码实现（片段）
## 实施计划
## 风险点与验证方式
## 待确认问题
```

**禁止事项**：
- ❌ 用户要求先审查时，不要直接修改代码
- ❌ 不要跳过方案直接实施

---

### 3. 实施代码

**后端规范**（如无项目规范，使用默认）：
```markdown
技术栈：.NET 10 + EF Core 10 + PostgreSQL 15

分层架构：
- API 层：HTTP 相关处理（HttpContext、Controller）
- Application 层：业务流程编排
- Domain 层：核心业务逻辑
- Infrastructure 层：数据持久化

编码规范：
- 使用主构造函数
- 使用文件范围 namespace
- DTO 使用 record 类型
- 异步方法名以 Async 结尾
- 实体使用充血模型
```

**前端规范**（如无项目规范，使用默认）：
```markdown
技术栈：Angular 21 + PrimeNG 21 + Tailwind CSS 4

目录结构：
- src/app/features/{module}/：业务模块
- src/app/shared/：全局共享组件
- src/_mock/：Mock 服务

编码规范：
- 使用 inject() 进行依赖注入
- 优先使用 PrimeNG 组件
- 使用 Tailwind CSS 布局
- 展示型组件使用 OnPush
- 使用 Signals 管理状态
```

---

### 4. 规范审查（内置，强制）

**时机**：代码编写完成后，构建验证之前。

**范围**：只做规范检查清单审查，不做需求对齐检查。

**与独立 code-review 的边界**：
| 检查项 | coding 内置审查 | 独立 code-review |
|--------|----------------|------------------|
| 规范检查清单审查（P0/P1/P2） | ✅ | ✅ |
| 自动化工具检查（dotnet format/eslint） | ✅ | ✅ |
| Must have 覆盖检查 | ❌ | ✅ |
| 策略偏离检查 | ❌ | ✅ |
| 文件遗漏检查 | ❌ | ✅ |
| 风险对策检查 | ❌ | ✅ |

**适用场景**：
- ✅ Phase 3 代码实现完成后自动执行
- ✅ 开发过程中的规范检查
- ❌ 用户要求「代码审查」→ 使用独立 code-review
- ❌ PR 提交前审查 → 使用独立 code-review
- ❌ 非本次开发的代码 → 使用独立 code-review

**流程**：
```markdown
1. 读取项目规范文档附录的“规范检查清单”
   → 后端：{project}/docs/standards/code-standard/backend-develop.md
   → 前端：{project}/docs/standards/code-standard/frontend-develop.md

2. 逐条对照本次代码变更
   → 每项标注 ✅ 通过 或 ❌ 违规

3. 如有 ❌，先修复再进入验证步骤
   → P0（违反强制规则）：必须修复
   → P1（违反推荐规则）：尽量修复
   → P2（风格建议）：记录即可

4. 输出检查结果摘要
```

**降级**：项目无规范文档时，使用本 Skill 模板的默认规则（见下方参考模板）。

**注意**：规范审查与构建验证是两个独立步骤。规范审查检查“是否符合规范”，构建验证检查“是否能编译通过”。

### 5. 验证

**后端验证**：
```markdown
1. 构建检查：dotnet build
2. 类型检查：编译通过
```

**前端验证**：
```markdown
1. 构建检查：ng build
2. 类型检查：编译通过
3. 浏览器验证：启动项目，验证主要路径
```

**注意**：测试运行（`dotnet test` / `ng test`）由 Phase 4 的 `test-runner` Skill 统一执行。

## ✅ 最佳实践

### 后端开发

1. **分层清晰**：HTTP 细节留在 API 层
2. **充血模型**：实体包含业务方法
3. **异步编程**：所有 I/O 操作使用 async/await
4. **DTO 映射**：使用 AutoMapper
5. **异常处理**：使用框架提供的异常类型

### 前端开发

1. **组件库优先**：优先使用 PrimeNG 官方组件
2. **单一职责**：每个组件只关注一个功能点
3. **单向数据流**：input 向下，output 向上
4. **Mock 同步**：Mock 数据与真实接口结构一致
5. **浏览器验证**：前端变更必须启动项目验证

## 🔗 完成衔接

**代码开发完成后，自动通知派发方（小架）。**

小架作为执行者，编码任务，完成后通过 completion event 自动回报小架。小架根据任务编排决定下一步：

| 执行结果 | 小架决策 | 下一动作 |
|-------------|---------|----------|
| ✅ 开发+审查+构建全部通过 | 继续编排 | 执行测试验证 |
| ❌ 规范审查有 P0 违规 | 阻塞 | 小架自行修复后重新执行 |
| ❌ 构建失败 | 阻塞 | 小架自行修复后重新执行 |

**部署由小架在测试通过后统一执行。**
