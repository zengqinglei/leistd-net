# AI Native 开发模式

## 核心理念

AI Native 开发不是让 AI 直接替代所有决策，而是让 AI 成为“可读规范、可追溯需求、可验证结果”的执行者。

## 协作闭环

```text
人类提出目标
  ↓
AI 澄清需求并生成 Plan
  ↓
人类确认边界和风险
  ↓
AI 小步实现
  ↓
AI 执行测试与自检
  ↓
人类验收或要求修正
```

## 角色分工

| 角色 | 主要职责 |
| --- | --- |
| 人类负责人 | 业务目标、优先级、验收、风险审批 |
| 规划 Agent | 澄清需求、拆解任务、维护 Plan 与登记册 |
| 开发 Agent | 读取规范、实现代码、同步模块文档 |
| 测试 Agent | 设计测试、执行验证、输出风险 |
| 审查 Agent | 依据规范检查缺陷、回归风险和遗漏测试 |

## 必须人工确认的事项

- 删除数据、迁移数据、修改生产配置。
- 引入新外部服务、费用、权限或合规风险。
- 修改认证授权、加密、审计、安全边界。
- 部署上线、回滚、重启生产服务。
- 使用真实密钥、账号、客户数据。

## 文档即上下文

AI 执行前应优先读取项目文档，而不是依赖历史对话：

- `docs/standards/`：长期规则。
- `docs/requirements/`：当前需求和验收标准。
- `docs/modules/`：模块边界、设计、API、测试。
- `docs/deploy/`：环境和发布约束。
- `docs/reviews/`：历史审查结论。

## 质量原则

- 需求必须可追溯到登记册。
- 设计必须说明取舍和风险。
- 实现必须遵守编码规范。
- 测试必须覆盖验收标准。
- 未完成或未验证事项必须显式记录。
