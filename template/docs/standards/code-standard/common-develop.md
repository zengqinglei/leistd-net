# 项目通用开发规范

> 本文是语言和技术栈无关的最低开发规范。具体 .NET、Angular 或其他技术栈规则应放在对应项目规范中；当项目没有技术栈定义时，AI 只应用本文原则和现有代码风格。

## 1. 开发原则

- **范围受控**：只实现已确认 Plan 范围；范围变化先记录并确认。
- **可读优先**：命名清晰、结构直接，避免过度抽象。
- **依赖最小化**：仅在没有合理替代方案时引入新依赖，并说明收益、成本和风险。
- **错误可诊断**：失败路径应有明确错误信息、日志或返回结果。
- **安全默认**：不硬编码密钥、Token、密码或真实客户数据；敏感信息必须脱敏。
- **文档同步**：新增 API、架构变化、部署变化或重要约定必须同步相关文档。

## 2. 变更规则

- 修改已有代码前先理解邻近实现和项目配置。
- 保持与现有目录、命名、格式、错误处理和测试风格一致。
- 删除、迁移或重命名代码必须有明确依据，并说明影响范围。
- 影响外部接口、数据结构、权限、安全、部署或费用的变更必须先确认。
- 不回滚用户未授权的变更。

## 3. 验证规则

- 优先运行项目声明的 build、lint、typecheck、test 命令。
- 只运行与当前阶段匹配的最小必要验证；完整测试由测试阶段执行。
- 测试失败时记录命令、错误摘要、影响和建议，不删除有效断言来制造通过。
- 未执行的验证必须说明原因，不能写成已通过。

## 4. Git 与提交

项目有 Git/分支/提交规范时以项目规范为准。无项目规范时建议使用 Conventional Commits：

```text
<type>(<scope>): <subject>
```

常用 type：`feat`、`fix`、`docs`、`style`、`refactor`、`test`、`chore`。

## 附录：规范检查清单

- [ ] 变更范围与 Plan 一致。
- [ ] 命名、目录和代码风格与邻近实现一致。
- [ ] 无硬编码密钥、真实账号或敏感数据泄露。
- [ ] 错误处理、日志和边界条件可诊断。
- [ ] 新依赖、新外部服务或高风险变更已说明并确认。
- [ ] 必要的文档、测试或配置已同步。
- [ ] 已执行项目声明的最小验证，或已说明未执行原因。
