# 测试规范

## 1. 文档边界

本文定义测试类型、目录、命名、执行命令、失败处理和测试报告格式。具体项目可在本规范基础上补充技术栈专用命令。

## 2. 测试类型和覆盖率

| 类型 | 目标 | 触发时机 |
| --- | --- | --- |
| 单元测试 | 验证函数、类、组件的局部行为 | 每次实现 |
| 集成测试 | 验证模块间、数据库、外部服务适配 | 涉及接口或持久化 |
| E2E 测试 | 验证核心用户路径 | 关键流程和发布前 |
| 回归测试 | 验证历史缺陷不复现 | 缺陷修复后 |
| 安全测试 | 验证认证、授权、输入、敏感信息 | 权限和外部接口变更 |

默认要求：

- 核心领域逻辑必须有单元测试。
- API 必须覆盖成功、失败、越权、边界输入。
- UI 关键路径至少有 E2E 或明确手工验证步骤。

## 3. 后端测试规范

### 3.1 目录

```text
backend/
├── tests/
│   ├── UnitTests/
│   ├── IntegrationTests/
│   └── TestSupport/
```

### 3.2 命名

- 测试类：`{ClassName}Tests`
- 测试方法：`Should_{ExpectedBehavior}_When_{Condition}`
- 测试数据构造器：`{Entity}TestDataBuilder`

### 3.3 要求

- 领域规则、权限判断、状态流转必须可测试。
- 数据库集成测试使用隔离数据库或事务回滚。
- 外部服务使用 fake/mock 或测试环境。
- 不使用生产密钥和真实客户数据。

### 3.4 命令

```bash
dotnet test
```

如项目拆分多个解决方案，应在 README 或本文件中声明具体命令。

## 4. 前端测试规范

### 4.1 目录

```text
frontend/
├── src/app/**/*.spec.ts
├── e2e/                     # 如项目启用 E2E
└── _mock/                   # Mock 数据与 API
```

### 4.2 命名

- 组件测试：`{component}.spec.ts`
- 服务测试：`{service}-service.spec.ts`
- Pipe 测试：`{pipe}-pipe.spec.ts`

### 4.3 要求

- service、pipe 和复杂业务函数必须有单元测试。
- 共享组件应覆盖输入、输出、错误态和空态。
- 核心表单应覆盖校验、提交、失败提示。
- HTTP 调用应使用 mock，避免依赖真实后端。

### 4.4 命令

```bash
npm test
npm run test
npm run e2e
```

项目应在 `package.json` 中声明实际可用命令。

## 5. E2E 测试规范

### 5.1 目录

```text
e2e/
├── specs/
├── fixtures/
└── reports/
```

### 5.2 必测核心流程

- 登录/认证状态。
- 关键资源的创建、查询、更新、删除。
- 关键表单校验。
- 权限不足和未登录处理。
- 关键错误态、空态、加载态。

### 5.3 命令

```bash
npm run e2e
```

## 6. 测试数据规则

- 使用可重复构造的测试数据。
- 不依赖执行顺序。
- 不写入生产环境。
- 敏感字段使用假数据。
- E2E 测试结束后清理可变数据。

## 7. 失败处理

测试失败时必须记录：

- 失败命令。
- 失败用例。
- 失败原因或日志摘要。
- 是否阻塞交付。
- 下一步处理建议。

AI 不得删除有效断言来“修复”测试失败，也不得声称通过未执行的测试。

```markdown
| 集成测试 | 通过/失败/未执行 | {说明} |
| E2E | 通过/失败/未执行 | {说明} |

## 3. 验收映射

| 验收项 | 验证方式 | 结果 | 证据 |
| --- | --- | --- | --- |

## 4. 风险与未覆盖项

```
