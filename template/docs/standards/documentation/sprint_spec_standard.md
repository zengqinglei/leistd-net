# Sprint Spec 文档标准

本文档定义 Spec 驱动开发的 **三阶段工作流程**、**目录与文件规范** 和 **文档内容模板**。

---

## 🌊 三阶段工作流程

所有功能开发 **必须严格遵循** **`需求收集 → 技术设计 → 任务规划`** 的三阶段流程：

### 阶段一：需求收集

- **目标**: 将粗略的功能想法转化为结构化的、可验证的需求文档，明确"做什么。
- **核心活动**:
  - 与产品、设计等相关方沟通，明确功能背景、目标和范围。
  - 编写用户故事 (User Stories) 来描述用户期望。
  - 使用 EARS 格式定义清晰、无歧义的验收标准 (Acceptance Criteria)。
  - 覆盖边界情况和异常场景。
- **输出**: 需求文档（`requirements.md`）
- **审批**: 需求文档需通过评审后，方可进入技术设计阶段。

### 阶段二：技术设计

- **目标**: 基于已确认的需求，创建全面、可行的技术设计方案，明确"怎么做。
- **核心活动**:
  - 进行必要的技术调研和现有代码分析。
  - 设计系统架构、组件、接口和数据模型。
  - 制定错误处理和测试策略。
- **输出**: 设计文档（`design.md`）
- **审批**: 技术设计方案需通过开发团队和架构师的评审，方可进入任务规划阶段。

### 阶段三：任务规划

- **目标**: 将复杂的设计方案分解为一系列可执行、可跟踪的编码任务，明确"如何分步完成。
- **核心活动**:
  - 将技术设计分解为原子化的、可独立执行的编码任务。
  - 明确每个任务的实现目标和验收条件。
  - 根据依赖关系和优先级对任务进行排序。
- **输出**: 任务文档（`tasks.md`）
- **确认**: 任务列表需获得开发团队的确认。

## 📁 目录与文件规范

### 目录结构

```
docs/sprints/YYYY/MMspX/{feature-name}/
  ├── requirements.md
  ├── design.md
  └── tasks.md
```

### 命名约定

- **年份 (`YYYY`)**: 使用四位数的年份，例如 `2025`。
- **Sprint (`MMspX`)**: `MM` 代表月份，`spX` 代表该月的第几次 Sprint。
  - `sp1`: 当前日期 ≤ 15日
  - `sp2`: 当前日期 > 15日
  - **注意：每月只有两个Sprint，不存在sp3、sp4等**
- **功能名 (`{feature-name}`)**: 必须使用 `kebab-case` (短横线分隔命名法)，例如 `user-authentication`。

## 📝 文档内容模板

### 需求文档 (`requirements.md`)

```markdown
# [功能名称] - 需求文档

## 1. 背景与目标
<!-- 简要描述此功能的业务背景、要解决的问题以及期望达成的业务目标。 -->

## 2. 用户故事
<!--
- 作为 [用户角色], 我想要 [实现某个功能], 以便 [获得什么价值]。
-->

- 作为 [角色], 我想要 [功能], 以便 [价值]

_（可根据需要添加更多用户故事）_

## 3. 核心功能点
<!-- 使用列表详细描述功能需求，每个点应清晰、可验证。 -->

- **功能点 A**:
  - **描述**: ...
  - **验收标准**:
    - [ ] 当 [特定前置条件] 时, 系统应 [展现出什么行为]。
    - [ ] 如果 [某个边界条件] 发生, 系统应 [如何正确处理]。
    - [ ] 当 [某种错误发生] 时, 系统应 [如何响应，例如显示错误提示]。

## 4. 非功能性需求
<!-- 描述性能、安全、兼容性等方面的要求。 -->

- **性能**: 页面加载时间应少于 2 秒。
- **安全**: 用户密码必须经过加密存储。

## 5. 范围之外
<!-- 明确列出本次迭代“不做”什么，以避免范围蔓延。 -->

```

```markdown

## 1. 方案概述

[简要复述核心需求，并概述你提议的技术方案。解释为什么选择这个方案，以及它如何满足需求。]

## 2. 架构设计
<!--
- 使用 Mermaid 图表或其他形式，展示新功能与现有系统的关系。
- 描述涉及的主要模块、组件或服务，以及它们之间的交互方式（例如 API 调用、事件传递）。
-->

## 3. 组件/模块设计

### [组件/模块 A]
- **核心职责**: ...
- **对外接口 (API)**:
  - `GET /api/v1/example`: 获取示例数据。
    - **请求参数**: ...
    - **成功响应 (200)**: ...
    - **失败响应 (4xx/5xx)**: ...
- **数据结构**:
  <!-- 描述此组件处理的核心数据模型。 -->

- **关键逻辑**:
  <!-- 描述复杂的算法、状态转换或核心业务流程。 -->

_（可根据需要添加更多组件/模块设计）_

## 4. 数据库设计
<!-- 如果涉及数据库变更，请提供表结构定义。 -->

**表名: `examples`**

| 字段名 | 类型 | 约束 | 描述 |

## 5. 风险与考量
<!-- 列出潜在的技术风险、依赖关系或需要进一步讨论的问题。 -->
```

### 任务文档 (`tasks.md`)

此文档的核心是将设计方案拆解为可执行的、独立的编码任务。

```markdown
# [功能名称] - 实施任务

## 1. [功能分类]
- [ ] **任务 1.1**: [清晰、可操作的任务描述]
  - [可选] 任务的某个具体子步骤或检查点。
  - _关联需求: [引用自 requirements.md 的具体需求点]_
- [ ] **任务 1.2**: [另一个清晰、可操作的任务描述]
  - _关联需求: [引用自 requirements.md 的具体需求点]_

## 2. [另一个功能分类]
- [ ] **任务 2.1**: [清晰、可操作的任务描述]
  - [可选] 任务的某个具体子步骤或检查点。
- [ ] **任务 2.2**: [另一个清晰、可操作的任务描述]
  - _关联需求: [引用自 requirements.md 的具体需求点]_

_（任务应按逻辑依赖关系排序）_

```
