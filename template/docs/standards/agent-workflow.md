# Agent 工作流规范

## 1. 适用范围

本规范适用于 AI Agent 参与的需求分析、方案设计、编码实现、测试验证、代码审查和部署准备。

## 2. 文件路径与命名

| 类型 | 路径 | 说明 |
| --- | --- | --- |
| 需求登记册 | `docs/requirements/registry.md` | 需求状态、负责人、关联文档 |
| 需求 Plan | `docs/requirements/{req-id}-plan.md` | 单个需求的实施方案 |
| 模块文档 | `docs/modules/{module}/` | 模块设计、API、计划、测试、状态 |
| 项目规范 | `docs/standards/` | 长期有效规则 |
| 部署文档 | `docs/deploy/` | 环境、发布、回滚 |
| 审查报告 | `docs/reviews/{feature}-code-review.md` | 代码审查输出 |

需求编号：`REQ-YYYYMMDD-NNN`。文件名使用小写：`req-yyyymmdd-nnn-plan.md`。

## 3. 需求 Plan 五段闭环

```markdown
# REQ-YYYYMMDD-NNN - {需求名称}

## 1. 需求&背景&目标
## 2. 核心策略
## 3. 实施步骤
## 4. 风险及应对策略
## 5. 验收闭环
```

### 3.1 需求&背景&目标

必须包含：

- 背景：为什么做，当前问题是什么。
- 需求：做什么、谁使用、核心场景是什么。
- 目标：可衡量的交付结果。
- 范围边界：本次包含和本次不包含。

### 3.2 核心策略

必须包含：

- 总体策略：怎么做，为什么这样做。
- 调整内容树形目录结构。
- 核心架构图或流程图。
- 重点注意事项及策略。
- 需要人工确认的风险或决策。

### 3.3 实施步骤

每个步骤必须包含：

| 步骤 | 实施内容 | 产出物 | 验证方式 |
| --- | --- | --- | --- |
| 1 | {内容} | {产出} | {验证} |

同时必须提供文件变更清单：

| 文件 | 操作 | 说明 |
| --- | --- | --- |
| `{path}` | 新增/修改/删除 | `{说明}` |

### 3.4 风险及应对策略

| 风险 | 概率 | 影响 | 应对策略 | 触发后的处理 |
| --- | --- | --- | --- | --- |
| `{risk}` | 高/中/低 | 高/中/低 | `{mitigation}` | `{fallback}` |

### 3.5 验收闭环

必须包含：

- Must have：必须满足，3-7 个。
- Nice to have：可选增强，0-3 个。
- 验收映射：每个 Must have 对应实施步骤、验证方式和证据。
- 完成定义：代码、审查、测试、构建、部署、用户验收。

验收映射格式：

| 验收项 | 对应实施步骤 | 验证方式 | 证据 |
| --- | --- | --- | --- |
| {验收项} | 步骤 1 | 测试/构建/API/页面验证 | {证据} |

完成定义：

- [ ] 代码实现完成
- [ ] code-review 无 P0 问题
- [ ] 测试通过或未执行原因已说明
- [ ] 构建通过或未执行原因已说明
- [ ] 文档同步完成
- [ ] 验收标准全部对照完成

## 4. 标准流程

1. **登记需求**：在 `registry.md` 中新增条目。
2. **读取上下文**：读取 `docs/standards/`、相关 `docs/modules/` 和历史审查报告。
3. **生成 Plan**：明确目标、范围、策略、风险和验收闭环。
4. **人工确认**：高风险变更必须确认后执行。
5. **拆解任务**：从实施步骤拆分可执行任务。
6. **小步实现**：优先最小可验证改动。
7. **同步文档**：代码影响模块/API/部署时同步更新文档。
8. **质量验证**：执行测试或说明无法执行原因。
9. **状态更新**：更新登记册和模块状态。

## 5. 状态定义

| 状态 | 含义 |
| --- | --- |
| candidate | 候选需求，尚未排期 |
| planned | 已形成 Plan，等待确认或排期 |
| in-progress | 正在实施 |
| blocked | 被外部依赖或决策阻塞 |
| review | 待审查或验收 |
| done | 已完成并通过验收 |
| archived | 已归档，不再活跃 |

## 6. 需求登记册更新规则

- 新需求进入前必须写入 `docs/requirements/registry.md`。
- 状态变化时同步更新登记册。
- 完成后记录完成时间、验收人和关联文档。
- 阻塞时记录阻塞原因、需要谁处理、下一步。

## 7. AI 执行约束

- 不得在未确认时执行生产部署、删除数据或修改密钥。
- 不得将具体项目私密信息写入模板文档。
- 不得跳过规范读取直接实现复杂需求。
- 不得声称执行了未执行的测试。
- 遇到需求冲突时，应先记录假设并请求确认。
- 修改已有代码时，不得回滚用户未授权的变更。

## 8. 高风险人工确认点

以下事项必须人工确认：

- 数据库迁移、批量更新、删除数据。
- 修改认证、授权、加密、审计、安全边界。
- 生产部署、回滚、重启服务。
- 引入新外部服务、费用、权限或合规风险。
- 使用真实密钥、账号、客户数据。
