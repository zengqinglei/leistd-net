# 账户中心设计原则

## 1. 背景

本轮围绕 Header 入口、个人信息编辑、头像上传、密码修改、前后端契约统一和 Mock 联调，逐步沉淀出一套适用于当前项目的账户中心设计原则。

这份文档的目标不是记录某一次具体实现，而是沉淀后续类似功能的判断依据，确保前端交互、后端分层、字段命名、校验边界和联调策略保持一致。

---

## 2. 核心原则

### 2.1 以用户操作链路为中心

账户中心相关功能优先围绕真实用户操作链路设计，而不是围绕字段堆叠设计。

本轮拆解出的核心链路包括：

1. 当前用户信息查询
2. 当前用户资料修改
3. 当前用户密码修改

因此交互上采用 Header 菜单作为统一入口，并拆分为两个独立 dialog：

1. 个人信息 dialog
2. 修改密码 dialog

这样可以避免资料编辑与安全操作混杂。

---

### 2.2 优先复用当前项目风格和组件体系

设计和实现应优先遵循当前项目的视觉语言、交互习惯和技术栈约束。

本轮落实为：

1. 入口保留在现有 Header 菜单体系中
2. 优先使用 PrimeNG 官方组件
3. 组件交互遵循当前项目轻量弹窗风格
4. 前端实现遵循当前 Angular 工程结构

这条原则要求新功能优先“嵌入现有系统”，而不是额外创造新页面范式。

---

### 2.3 先统一契约，再分别实现

前端、Mock、后端必须先统一字段命名和接口语义，再分别开发。

典型体现：

1. `avatarUrl` 统一收敛为 `avatar`
2. `GET /api/v1/auth/me`、`PUT /api/v1/auth/me`、`POST /api/v1/auth/change-password` 三条链路统一定义
3. `frontend/_mock` 与正式后端接口保持一致

这条原则用于避免：

1. 前端能提交但后端不接受
2. Mock 可用但真实接口不一致
3. DTO、实体、前端模型命名割裂

---

### 2.4 命名表达业务语义，不表达临时实现

字段命名优先表达业务含义，而不是表达当前存储介质或一次性实现方式。

典型体现：

1. `avatar` 不绑定 URL
2. 当前头像既可存 base64，也可扩展为 URL 或对象存储地址

因此命名原则为：

1. API 命名面向未来演进
2. 领域模型命名优先保持稳定
3. 实现细节不应直接泄露到业务契约里

---

### 2.5 校验分层明确

校验必须按职责分层，而不是在各层重复堆叠。

#### 前端

负责体验型校验：

1. 必填
2. 长度
3. 格式
4. 两次输入是否一致
5. 文件大小与类型约束

#### DTO

负责接口级输入约束：

1. 数据注解
2. 基础长度限制
3. 基础格式限制

#### Domain

负责真正业务规则：

1. 用户名唯一性
2. 邮箱唯一性
3. 当前密码是否匹配
4. 当前用户是否允许修改本地密码

这条原则要求：

1. 不在领域层重复 DTO 已处理的联动校验
2. 不把业务唯一性校验错误地放到前端
3. 前端校验用于优化体验，后端校验用于保证正确性

---

### 2.6 应用层只做编排，不承载领域真规则

Application Service 的职责是组织流程，而不是承载真正业务规则。

本轮收敛后的应用层职责包括：

1. 获取当前用户
2. 调用领域服务
3. 持久化
4. DTO 映射

不应在应用层重复做：

1. 领域唯一性校验
2. 密码规则判断
3. 多余的字符串清洗包装

---

### 2.7 领域逻辑尽量收口到领域对象或领域服务

密码哈希、创建用户、修改密码等属于领域行为，应尽量由领域层统一封装。

本轮沉淀出的原则是：

1. 账号创建能力优先收口到 `UserDomainService`
2. 密码哈希不在多个应用服务里散落调用
3. 薄包装领域服务应及时删除，避免形成无意义中间层

---

### 2.8 Mock 不是假数据，而是联调契约镜像

`frontend/_mock` 的作用是支撑完整联调，不是只返回“看起来能跑”的假响应。

因此 Mock 必须做到：

1. 接口路径与正式接口一致
2. 字段结构与正式 DTO 一致
3. 修改后能够联动刷新当前用户状态
4. 支撑查询、更新、修改密码完整链路

---

### 2.9 交互设计以就地完成为优先

用户进入账户中心后，应尽量在当前弹窗内完成任务，而不是频繁跳转页面。

具体体现：

1. 资料修改在个人信息 dialog 内完成
2. 头像在当前展示区域 hover 后直接上传
3. 选择文件后在当前组件内立即预览
4. 密码修改独立 dialog 完成，不跳转设置页

---

### 2.10 先保证主链路可用，再收紧细节一致性

实现顺序应遵循：

1. 先打通入口和核心提交流程
2. 再对齐命名
3. 再优化校验提示
4. 再收敛领域边界
5. 最后清理编译 warning 和冗余结构

这条原则适合多轮迭代式开发，而不是一次性试图把所有层全部做满。

---

## 3. 设计与实现方法论

基于本轮实践，后续类似功能建议统一采用以下方法：

1. 先识别真实业务链路
2. 先统一接口契约与命名
3. 前端先完成入口、交互和状态联动
4. Mock 同步补齐
5. 后端按 Controller -> AppService -> Domain -> Entity -> Persistence 分层实现
6. 校验按体验、接口、业务三层分配
7. 每轮迭代优先收敛职责边界
8. 命名面向未来扩展，不绑定临时实现

---

## 4. 本轮典型结论

### 4.1 交互层

1. 个人信息与修改密码必须分开
2. 头像上传适合就地 hover 交互
3. 官方组件优先于自定义拼装

### 4.2 契约层

1. 用户模型统一使用 `avatar`
2. `auth/me` 是当前用户资料的统一入口
3. Mock 必须同步更新

### 4.3 分层层

1. DTO 处理输入约束
2. AppService 处理流程编排
3. Domain 处理业务规则
4. Entity 处理自身状态变更

### 4.4 命名层

1. 业务字段优先稳定
2. 存储方式可变，字段语义应稳定

---

## 5. 后续适用范围

这份原则适用于后续所有与“当前登录用户账户中心”相似的功能，包括但不限于：

1. 资料中心
2. 安全设置
3. 用户偏好设置
4. 渠道分组与订阅配置等需要前后端契约统一、交互收敛和职责清晰的功能

后续如需新增能力，应优先回看这份原则，而不是从零重新定义实现方式。
